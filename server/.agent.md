# Sage AI — Agent & Coding Rules

These rules apply to all AI-assisted and human contributions to the NestJS backend.
Violations must be fixed before a PR is merged.

---

## Architecture Rules

### Module Boundaries
- Every feature lives in its own NestJS module under `src/<feature>/`.
- A module must export only what other modules legitimately need (usually just the service).
- Never import a schema from another module's `schemas/` folder directly. Re-export via the module.
- `CommonModule` is the only module allowed to be globally imported without explicit feature coupling.

### File Layout (per module)
```
<feature>/
├── <feature>.module.ts       ← NestJS module decorator, imports/exports
├── <feature>.controller.ts   ← Route handlers ONLY — no business logic
├── <feature>.service.ts      ← All business logic, DB calls
├── schemas/
│   └── <entity>.schema.ts    ← Mongoose Schema + Document type
└── dto/
    └── <action>-<entity>.dto.ts  ← class-validator DTOs
```

### Naming Conventions
- Files: `kebab-case.ts`
- Classes: `PascalCase`
- Methods/variables: `camelCase`
- MongoDB collection names: `snake_case` (plural)
- Enum values: `UPPER_SNAKE_CASE`
- DTO suffix: always `Dto` (e.g., `CreateEmployeeDto`, `LoginDto`)
- Schema suffix: always `Schema` (e.g., `EmployeeSchema`)
- Guard suffix: always `Guard` (e.g., `JwtAuthGuard`, `RolesGuard`)

## TypeScript Rules

- No `any` types — use `unknown` and narrow, or define proper interfaces.
- All service methods must have explicit return type annotations.
- Use `Partial<T>` for update DTOs rather than repeating all fields as optional.
- Prefer `interface` for plain data shapes; use `class` only for DTOs and schemas.
- All async functions must be declared `async` and use `await` — no raw Promise chains.

---

## NestJS-Specific Rules

### Controllers
- Controllers are route handlers only. They must not contain any business logic.
- Always annotate the controller with `@ApiTags()` for Swagger grouping.
- Always annotate route params/bodies with `@ApiOperation()` and `@ApiResponse()`.
- Use `@HttpCode(HttpStatus.XXX)` explicitly when the default 200/201 is wrong.
- Apply guards at the controller or method level — never inside a service.

### Services
- Services must be `@Injectable()`.
- Never call `process.env` directly — always inject and use `ConfigService`.
- Services may inject other services only if the dependency is exported by that feature's module.
- Throw `NotFoundException`, `BadRequestException`, etc. — never return `null` to indicate failure.

### DTOs
- Every DTO property must have at least one `class-validator` decorator.
- Use `@IsOptional()` explicitly for optional fields; do not rely on TypeScript optionality alone.
- Use `@Transform()` for sanitisation (e.g., `trim`, `toLowerCase`) before validation runs.
- Nest DTOs for complex nested payloads (e.g., `SalaryAmountDto[]` inside `UpdateScheduleDto`).

### Schemas
- Every schema property must have `@Prop()` with explicit `type`, `required`, and `default` where applicable.
- Add `{ timestamps: true }` to all `@Schema()` decorators.
- Use `Types.ObjectId` (not `string`) for cross-collection references.
- Never store plain-text passwords or API keys — hash passwords with bcrypt; encrypt API keys with AES.
- Mask sensitive fields in responses: account numbers show last 4 digits only; phone numbers masked.

### Guards & Decorators
- `JwtAuthGuard` must be applied to every route that requires authentication.
- `RolesGuard` must always be used together with `@Roles()` — never one without the other.
- `@Public()` decorator opts a route out of global JWT guard (used only for health check, webhook, verify endpoints).
- The `@CurrentUser()` decorator extracts the authenticated user from the JWT payload — use it instead of `@Req()`.

## Security Rules

- Never log sensitive data (tokens, passwords, API keys, GPS coords, account numbers).
- Validate Squad webhook HMAC signature before processing any webhook payload.
- Employee deep-link tokens must be single-use: mark `isConsumed = true` on first successful submit.
- All admin-mutating endpoints (freeze, hold, import) require `hr_admin` role.
- All auditor-investigation endpoints require `auditor` role.
- Apply `ThrottlerGuard` globally to prevent brute-force on `/auth/login`.
- Helmet headers enabled globally in `main.ts`.

---

## Database Rules

- Always use Mongoose lean queries (`.lean()`) for read-only operations that don't need Mongoose document methods.
- Paginate all list endpoints — default page size 20, max 100.
- Index all fields used in `find()` filters: `orgId`, `employeeId`, `status`, `cycle`.
- Never delete records permanently — use soft-delete patterns (`deletedAt` timestamp).

---

## Testing Rules

- Unit tests live at `<feature>/<feature>.service.spec.ts`.
- E2E tests live in `test/`.
- Mock Mongoose models with `@nestjs/testing` and `getModelToken()`.
- Never test implementation details — test observable behaviour (return values, thrown exceptions, called dependencies).

---

## Environment Variable Rules

All env vars are typed through `config/configuration.ts`. Do not add new env vars without:
1. Adding the key to `.env.example` with a description comment.
2. Adding it to the `configuration()` factory function.
3. Using it via `ConfigService.get<T>('namespace.key')`.

---

## PR Checklist

Before marking a PR ready for review:
- [ ] No `any` types introduced
- [ ] All new routes have DTOs with validation
- [ ] All new routes have `@ApiOperation` / `@ApiResponse` annotations
- [ ] New sensitive fields are masked in response
- [ ] Guards applied to all new protected routes
- [ ] `.env.example` updated if new env vars added
- [ ] Unit test exists for new service methods
